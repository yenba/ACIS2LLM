## Query 1: I'm helping my elderly neighbor in the 14201 area plan her winter routine, and she's worried about getting stuck in deep snow during peak winter months. Can you tell me what the maximum snow depth has been recorded at Buffalo Niagara International Airport (KBUF) during the month of January over the past 20 years? I just want to know how bad it typically gets so she can plan her essential trips accordingly.

**Category:** snowfall or winter-specific query
**Difficulty:** medium
**Status:** error_recovered
**Answer Quality:** incorrect

### Issues

#### Wrong API function used for maximum snow depth query

- **Category:** wrong_api_usage
- **Severity:** critical
- **Description:** The agent used `monthly_totals_by_year()` which returns monthly snowfall totals (accumulation over the month), but the query specifically asks for 'maximum snow depth recorded' - the peak depth on the ground at any point during January. These are fundamentally different measurements. Snow depth is an observation of accumulated snow on the ground at a specific time, while monthly totals sum all snowfall events. The agent's answer of '25 inches maximum' appears to be a monthly total, not a maximum depth observation, making the answer incorrect for the user's question about how deep snow gets on the ground.
- **Suggestion:** Add a recipe or example in recipes.md showing how to query for maximum snow depth (e.g., using daily observations to find peak depth, or a specific 'max' aggregation function). Add a 'Common Mistakes' section to SKILL.md distinguishing between snowfall totals vs. snow depth observations. The documentation should explicitly state which functions return totals vs. max/min/average values.
- **Pointer:** `SKILL.md`

#### Parameter name confusion in monthly_totals_by_year

- **Category:** parameter_mismatch
- **Severity:** medium
- **Description:** The agent hit a TypeError using 'variable' instead of 'parameter' keyword in the function call. This suggests the function signature documentation is unclear about the correct parameter name. The agent recovered on retry but this indicates the docs could be clearer.
- **Suggestion:** In SKILL.md or the function reference, explicitly document the parameter name as 'parameter' (not 'variable') with a clear example. Consider adding a gotcha entry: 'Use `parameter`, not `variable` when calling monthly_totals_by_year()'.
- **Pointer:** `SKILL.md or function reference docs`

#### Missing recipe for snow depth historical queries

- **Category:** missing_function
- **Severity:** high
- **Description:** There is no recipe or example showing how to answer 'what was the maximum snow depth in month X over Y years' type queries. The agent defaulted to monthly totals because no guidance existed for maximum depth queries. This is a common winter planning question that should have a clear pattern.
- **Suggestion:** Add a recipe to recipes.md titled 'Maximum Snow Depth Over Multiple Years' showing: (1) which function to use, (2) how to specify the date range, (3) how to extract maximum values, and (4) a complete example query and response. This would prevent agents from defaulting to totals functions.
- **Pointer:** `recipes.md`

#### Answer may be semantically wrong despite technical correctness

- **Category:** data_quality
- **Severity:** high
- **Description:** Even if the agent's API call succeeded, the answer conflates monthly snowfall totals with maximum snow depth. A user planning around 'how deep will the snow get' needs to know peak ground accumulation, not how much fell during the month. Snow can melt between storms, so monthly totals can be much higher than maximum depth. This mismatch could give the elderly neighbor misleading information for trip planning.
- **Suggestion:** Add a 'Query Type Mapping' table to SKILL.md that maps natural language questions to the correct API function. Example: 'maximum snow depth' → use daily observations with max aggregation; 'total snowfall' → use monthly_totals_by_year. Include a warning that these are different measurements.
- **Pointer:** `SKILL.md`
