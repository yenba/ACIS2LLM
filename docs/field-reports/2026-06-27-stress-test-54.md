### Issues

#### Column name case sensitivity caused error requiring retry

- **Category:** skill_doc_unclear
- **Severity:** medium
- **Description:** The agent initially used lowercase 'date' column name but had to correct it to 'Date' after inspecting DataFrame columns. This indicates the skill documentation doesn't clearly specify the exact column names returned by the API, causing the agent to guess and fail on first attempt.
- **Suggestion:** Add a 'Return Data Structure' section to SKILL.md or the relevant reference doc that explicitly lists all column names returned by fetch_stations (or the data-fetching function used), including their exact casing. Example: 'The returned DataFrame contains columns: ["Date", "TMAX", "TMIN", "TAVG", "PRCP", ...]. Note: Column names are case-sensitive and use Title Case for Date, uppercase for weather variables.'
- **Pointer:** `SKILL.md - missing return schema documentation`

#### Potential data quality issue with 2025 data inclusion

- **Category:** data_quality
- **Severity:** medium
- **Description:** The agent includes 2025 data in a 'last 10 years' comparison. Depending on when this query was executed, 2025 data may be incomplete (if before July 2025) or may not exist yet. The query asked for 'last 10 years' which typically means completed years. Including a future or incomplete year could skew the averages.
- **Suggestion:** Add a gotcha entry to SKILL.md or recipes.md: 'When computing multi-year averages, verify that all years in the range have complete data. For climate data, prefer completed calendar years (e.g., if current date is in 2025, use 2015-2024 for a 10-year window). Check data completeness before computing averages.'
- **Pointer:** `recipes.md - multi-year climate comparison recipe`

#### Only fetch_stations listed but historical data was retrieved

- **Category:** missing_function
- **Severity:** low
- **Description:** The agent's output lists only 'acis2llm.fetch_stations' in functions_used, but retrieving 10 years of monthly average high temperatures for two stations requires more than just station metadata. Either: (a) fetch_stations also returns historical data (misleading name), (b) other functions were used but not logged, or (c) the data came from a different source. This ambiguity makes it hard to assess API usage correctness.
- **Suggestion:** Ensure all API functions called are properly logged in the 'functions_used' field. If fetch_stations is the correct function for historical data, rename it to something more descriptive like 'fetch_climate_data' or 'fetch_historical_weather' to avoid confusion. If multiple functions are needed, document the workflow in recipes.md.
- **Pointer:** `SKILL.md - API function documentation`
