**Category:** edge-case time period
**Difficulty:** medium
**Status:** clean
**Answer Quality:** correct

### Issues

#### Missing recipe for multi-year same-date queries

- **Category:** doc_gap
- **Severity:** medium
- **Description:** The agent successfully queried Feb 29 across 17 leap years, but there's no evidence of a recipe in the skill docs for this common pattern (querying the same calendar date across multiple years). The agent had to figure out independently that it needs to call get_single_station_acis_data 17 times (once per leap year: 1960, 1964, 1968, ..., 2024) and aggregate the results. Without a recipe, agents may struggle with this pattern or hallucinate statistics.
- **Suggestion:** Add a recipe in recipes.md titled 'Query same calendar date across multiple years' that shows: (1) how to identify the year range and specific dates, (2) the loop pattern for calling get_single_station_acis_data for each year, (3) how to aggregate results (calculate average, min, max, rank), and (4) error handling for years with missing data.
- **Pointer:** `recipes.md`

#### No explicit guidance on leap year/Feb 29 queries

- **Category:** doc_gap
- **Severity:** low
- **Description:** Feb 29 is an edge case date that only occurs in leap years. The skill docs likely don't mention this edge case explicitly. While the agent handled it correctly by identifying 17 leap years in the range, future agents might not realize they need to filter for leap years only.
- **Suggestion:** Add a gotcha entry in SKILL.md or references/ mentioning that Feb 29 queries require special handling: only leap years have this date, and agents should calculate which years in a range are leap years (divisible by 4, except centuries not divisible by 400). Include a code snippet showing leap year calculation.
- **Pointer:** `SKILL.md or references/`

#### functions_used reporting is ambiguous

- **Category:** other
- **Severity:** low
- **Description:** The agent reports functions_used as ['xmacis2py.get_single_station_acis_data'], which only shows the unique function name, not the count of calls. For a task requiring 17 API calls (one per leap year), this could be confusing for debugging. It's unclear if the agent actually made 17 calls or hallucinated the multi-year statistics.
- **Suggestion:** Consider updating the reporting format to show call counts (e.g., {'xmacis2py.get_single_station_acis_data': 17}) or add a separate field for total_api_calls. This would help verify that the agent actually performed the necessary queries.
- **Pointer:** `Skill code that generates the output JSON`
